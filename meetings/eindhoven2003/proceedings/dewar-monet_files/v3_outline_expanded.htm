Web Services

Web Services Overview
Web Service technologies allow:
Integration of computational components
Distribution of components (useful for pervasive computing / collaborative working / sharing hardware …)
Service providers / IP stakeholders to control exactly what they expose to clients
Key Technologies (WorldWide Web Consortium, …):
SOAP (XML-messaging)
WSDL (Low-level interface description)
Choreography languages (BPEL4WS, WSCI, …)
Implementations:
IBM Websphere, Microsoft .NET, BEA WebLogic, Sun JWSDP, TheMindElectric Glue, …

How will web services be deployed?
As resources within physical organisations
behind a firewall Þ security/authentication less of an issue
… within virtual organisations
authentication an issue
partners trust services to behave as advertised
partners have control of availability of services
As public service available to anybody
free/pay-per-use/subscription …
issues of trust
need framework for contracts, payments, dispute resolution …

Web Services Today

Web Service Description Language (WSDL)
XML application for describing how to bind to a service
A WSDL document defines:
Messages (i.e. data objects)
Operations (i.e. methods)
Ports (i.e. classes)
Complex data types described via XML Schema
WSDL document usually generated automatically
Client can generate proxy template from WSDL document

Web Service Discovery: Existing Technologies
Universal Description Discovery and Integration (UDDI)
Taxonomy-based
Original version not very nimble or dynamic
Evolving towards more flexibility
Web Service Inspection Language (WSIL)
Format for collecting links to service descriptions
e.g. pointer to entry in UDDI registry

Mathematics On the NET
MONET Project: 2 Year EU-funded RTD activity
Principal goals:
Design an architecture for brokering mathematical web services
Develop the necessary mechanisms for describing services, problems and queries
Build prototype brokers and services based on generic technologies
Partners:
NAG Ltd, Stilo Technology Ltd, Universities of Bath, Eindhoven, Manchester and Western Ontario, I3S Nice

Why Mathematical Web Services?
Mathematics underpins a large proportion of scientific and commercial activities
NAG software used in Engineering, Manufacturing, Finance, Utility Provision, Astrophysics, Education, …
Software increasingly complex and specialised
different languages for application and algorithm
application developers lack mathematical knowledge to identify most appropriate technologies
Mathematics is relatively well-defined so automating service discovery and choreography should be feasible

Simplified MONET Architecture

Service Registration

Client Submits Query

Execution Plan Created

Plan Returned To Client

Client Approves Plan, Requests Resources

Client Gains Access to Resources

Client Interacts With Service

Role of the Broker
Client Manager:
API for client
Service Manager:
Maintain service registry
Negotiate service access
Perform various housekeeping duties
Planning Manager:
Extract discrimination criteria from queries
Match to features of registered services
Construct one or more execution plans with indicators of adherence to client’s specification

Other MONET Components
Planning Services
Invoked by broker
Create execution plans
Domain- and possibly service-specific (e.g. NAG ODE planner)
Execution Services
Implement choreography languages
Mathematical Object Manager
Registry for mathematical objects
Can refer to an object by URI
Explanation service
Provide a human client with explanation of solution process
Commercial Infrastructure
Banks, resource usage services, service certification agencies, …

Conclusions
The web service architecture could underpin software integration and distribution in the future
widespread support in the IT industry
cross-platform, open standards
Semantic web is an attractive vision
not yet any accepted standards
little tool support
Mathematical service infrastructure feasible, but must
be layered on top of commodity technologies (SOAP, WSDL, …)
accommodate different kinds of mathematical service
co-exist with other mechanisms (OGSA, GESA, …)
not make service deployment disproportionately expensive
