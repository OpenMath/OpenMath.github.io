OpenMath Attributes

Issues
Want to be able to translate arbitrary MathML objects to OpenMath and back
Want to make use of XML tools for transformations, rendering etc.
Must remember that not all OpenMath applications handle arbitrary XML

MathML Semantics Example
<semantics>
  <apply>
    <plus/>
    <apply> <sin/> <ci> x </ci> </apply>
    <cn> 5 </cn>
  </apply>
  <annotation-xml encoding="MathML-Presentation"> ... ...
  </annotation-xml>
  <annotation encoding="TeX">\sin x+5</annotation>
  <annotation-xml encoding="OpenMath">
    <OMA xmlns="http://www.openmath.org/OpenMath">
      <OMS cd="transc1" name="sin"/>
      <OMI>5</OMI> </OMA>
  </annotation-xml>
</semantics>

Current Mechanism
Use altenc CD and
encode XML as a string
must escape < in XML encoding so XML tools can’t process it
this is technically illegal (standard says that XML-encoded OpenMath objects cannot include entity or character references)
encode XML as a bytearray
base64 encoded so XML tools can’t process it in XML encoding

Example
<OMOBJ>
 <OMATTR>
  <OMATP>
   <OMS cd="altenc" name="MathML_encoding"/>
   <OMSTR> &lt;sin&gt; 1.5 &lt;/sin&gt;</OMSTR>
   <OMS cd="altenc" name="LaTeX_encoding"/>
   <OMSTR> \sin(1.5) </OMSTR>
  </OMATP>
  <OMA>
   <OMS cd="transc1" name="sin"/>
   <OMF dec="1.5"/>
  </OMA>
 </OMATTR>
</OMOBJ>

Other “CD-based” approach
Use a “reference” CD to point to an object elsewhere in the document
<OMATP>
  <OMS name=“id” cd=“reference”/>
  <OMI>42</OMI>
</OMATP>
lose integrity of objects
not understood by XML tools

Proposals (Watt and So)
Introduce new <OMXML> element into OpenMath
only really makes sense in XML encoding
Extend OMATP:
allow “pseudo-cd” in first child
allow arbitrary XML data for second child
implications to be discussed later

Example
<OMATTR>
 <OMATP>
  <OMS cd=“xml” name=“MathML-Presentation”/>   <mrow>
    <mo>sin</mo>&ApplyFunction;<mn>1</mn>
  </mrow>
 </OMATP>
 <OMA>
   <OMS cd=“transc1” name=“sin”/>
   <OMI>1</OMI>
 </OMA>
</OMATTR>

Alternative Proposal (Dewar)
Introduce new OMDATA construct at object level with two fields:
encoding (e.g. “MathML-Presentation”)
value (e.g. <math> … </math> )
Advantages
No abuse of CD mechanism
Separation of semantics and encoding
Not XML-specific
Disadvantages
Major change

Example
<OMATTR>
 <OMATP>
  <OMS cd=“presentation” name=“visual”/>
  <OMDATA encoding=“MathML_Presentation”>
   <mml:mrow xmlns:mml=“http://www.w3c.org/1998/Math/MathML”>
    <mml:mo>sin</mml:mo>&#8289;<mml:mn>1</mml:mn>
   </mml:mrow>
  </OMDATA>
  <OMS cd=“presentation” name=“speech”/>
  <OMDATA encoding=“mp3”>
   <OMB> … … … </OMB>
  </OMDATA>
 </OMATP>
 <OMA> <OMS cd=“transc1” name=“sin”/> <OMI>1</OMI> </OMA>
</OMATTR>

Open Questions: Binary Encoding
“If OMDATA construct includes XML then encode it as an XML-equivalent string”
How do we express that in the standard?

Open Questions: XML Encoding
OpenMath has restrictions on (linearised) XML
UTF8
No character or entity references
Cannot encode e.g. <OMSTR></OMSTR> as <OMSTR/>
Restrictions on where comments and processing instructions can occur
Syntactic restrictions on element content, attribute values
No <!DOCTYPE allowed
Contents of OMDATA could be legal XML but not legal wrt OpenMath XML encoding
All OpenMath applications must support the XML encoding
How does an OpenMath application which does not support arbitrary XML parse the contents of OMDATA?
DTD can no longer validate

Possible Solutions
Support minimal set of XML entities in strings (&lt; &amp;) to allow altenc to work
minimal change
no XML tool support for manipulating embedded XML fragments
Allow any legal XML in XML encoding but only mandate that applications support (variant of) existing subset
would probably need extra error CD
could be great source of confusion!
Make XML encoding fully XML-compliant
best use of existing XML tools
all OpenMath-reading applications need full XML parser
probably requires new version of standard
